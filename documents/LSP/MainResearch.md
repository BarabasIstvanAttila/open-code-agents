Skills are the right primitive here because:
Skills are lazy-loaded via the skill tool ÔÇö they inject zero tokens until an agent recognizes it needs them. The same skill is composable across all agents: build loads esp32-idf when it needs to run idf.py, plan loads esp32-patterns when designing a FreeRTOS component, scout loads esp32-idf when interpreting LSP diagnostics. You'd never get that cross-agent reuse from a dedicated agent.
The two skills and what they cover:
esp32-idf is the toolchain skill ÔÇö environment setup, every idf.py command with the right flags, the compile_commands.json lifecycle, how to triage LSP false positives vs real errors, build error patterns with their fixes, monitor output decoding, and OpenOCD/GDB setup. This is the operational knowledge any agent needs to do ESP32 work.
esp32-patterns is the domain knowledge skill ÔÇö component CMakeLists structure, FreeRTOS task/queue/semaphore/event group patterns with correct stack sizing and priority guidelines, GPIO ISR rules, SPI and I2C driver patterns for both the old and new IDF APIs, NVS, memory region selection (DRAM/IRAM/DMA), safe C++ idioms under IDF constraints (RAII mutex guard, type-safe GPIO wrapper, the global init pitfall), and a common bugs cheatsheet. This is what the agent needs to design ESP32 code correctly.

What init.sh generates and why:
opencode.json is the central config. It wires five things together. The "lsp" block overrides OpenCode's built-in clangd (which auto-installs the wrong generic version) with the ESP-IDF-specific esp-clangd binary, passing the --query-driver glob that points to your cross-compiler so clangd can interrogate it for Xtensa/RISC-V include paths. Crucially, it also passes IDF_PATH via the "env" property directly to the clangd process ÔÇö so the LSP server has the right environment even if you opened OpenCode from a launcher that didn't source IDF. The --compile-commands-dir=build points it at the JSON database idf.py reconfigure produces.
The "shell" key points at .opencode/esp-shell.sh, which is a thin bash wrapper that sources export.sh before every command. This means every idf.py call the agent makes has the full IDF environment automatically ÔÇö no "idf.py: command not found" mid-session.
The "permission" blocks are scoped per-agent to your five custom agents (dev/qa/scout/plan/mem). scout and plan get edit: deny, bash: deny so they're truly read-only. qa gets edit: deny but full bash except flash/erase which require confirmation. dev gets everything except destructive operations. Skill permission is "*": "allow" globally, so any agent can load either skill without a prompt.
The "command" block gives you /esp-build, /esp-check, /esp-flash, /esp-size, /esp-debug, and /esp-new-component as slash commands in the TUI. Each one tells the agent which skill to load and exactly what to do.
The skill loading is automatic ÔÇö agents see <available_skills> in their tool list and choose esp32-idf or esp32-patterns based on the task. /esp-build explicitly tells the agent to load the skill in the template, but for freeform requests the agent decides on its own.
The most common reason things break: compile_commands.json goes stale after you add a new source file or component. Run /esp-check in OpenCode and the dev agent will reconfigure and report real errors vs clangd false positives.
