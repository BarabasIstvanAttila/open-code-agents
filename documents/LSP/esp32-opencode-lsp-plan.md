# ESP32 × OpenCode LSP — Setup & Agentic Coding Plan

> Target: Mac Studio M2 · ESP-IDF ≥ v5.x · OpenCode with `opencode-shared` agent architecture

---

## 1. What LSP Actually Does for the Agent

Language Server Protocol is a JSON-RPC bridge between a *language server* (a process
that deeply understands your code) and any *client* that speaks the protocol — in this
case OpenCode. When enabled, OpenCode sends every file it opens to the server and
receives back a stream of:

| LSP signal | What the agent gets |
|---|---|
| **Diagnostics** | Errors & warnings with exact file:line:col — fed as context into the agent's ReAct loop |
| **Hover / Definition** | Type info, doc comments, cross-file navigation — agent can resolve symbols without grep |
| **Completion** | Ranked suggestions — agent uses to fill in API calls correctly |
| **Code actions** | Automated fixes (missing includes, casts) — agent can apply without a full build |

The critical value for ESP32 is *diagnostic feedback without a full `idf.py build`*.
A build on a clean cache can take 3–5 minutes; clangd gives back include-path errors
and type mismatches in milliseconds. This dramatically tightens the agent's inner loop.

### When NOT to rely solely on LSP

The OpenCode docs are explicit: LSP can get out of sync and is not always a net
positive. For ESP32 specifically, the right split is:

- **LSP on** → symbol lookup, type errors, include errors during *editing*
- **`idf.py build` via CLI** → final truth for linker errors, partition issues, CMake
- **`idf.py flash && idf.py monitor`** → runtime / hardware truth

Document both in `AGENTS.md` so every agent knows which signal to trust.

## 2. Why ESP32 Needs Special Treatment

Standard `clangd` doesn't know about Xtensa or RISC-V cross-compilers. It will
flood the agent with thousands of false "not found" errors because it can't locate
ESP-IDF headers. Two things must be fixed:

1. **`--query-driver`** — tells clangd to interrogate the cross-compiler directly
   for its built-in include paths (sysroot, compiler-specific headers).
2. **`compile_commands.json`** — the compilation database that records the exact
   compiler flags used for every `.c`/`.cpp` file. Must exist *before* clangd starts.

ESP-IDF ships its own `esp-clang` toolchain (LLVM-based) that understands all the
ESP32 targets. Using it instead of the Homebrew `clangd` is strongly recommended.

```bash
# 4. OpenCode ≥ 0.x with lsp support (clangd is built-in, we override it)
opencode --version
```

## 4. Generate `compile_commands.json`

This file is the foundation of everything. Without it clangd is blind.

```bash
# Inside your ESP-IDF project root, with IDF environment sourced:
. $IDF_PATH/export.sh

# Standard GCC build (works, but compile_commands has GCC-specific flags clangd
# can't parse — we patch them with .clangd file):
idf.py reconfigure          # fast: only CMake, no compilation
# compile_commands.json is now at ./build/compile_commands.json

# Alternative: clang-native build (cleaner compile_commands, experimental):
idf.py -B build.clang -D IDF_TOOLCHAIN=clang reconfigure
# compile_commands at ./build.clang/compile_commands.json
```

> **Tip:** Add `idf.py reconfigure` to your `check.sh` health script so the
> database is always fresh when a new agent session starts.

## 5. Project-level `.clangd` Config

Create this at your ESP-IDF project root. It filters out GCC-specific flags that
clangd rejects and points it at the right build directory.

```yaml
# .clangd  (project root)
CompileFlags:
  # Point at the compile_commands.json generated by idf.py
  CompilationDatabase: build

  # Strip GCC-only flags that make clangd emit noisy errors
  Remove:
    - -mlongcalls          # Xtensa-specific
    - -mtext-section-literals
    - -fno-tree-switch-conversion
    - -fstrict-volatile-bitfields
    - -fno-jump-tables
    - -mfix-esp32-psram-cache-issue
    # For RISC-V targets, remove instead:
    # - -march=rv32imc
    # - -mabi=ilp32

Diagnostics:
  # Suppress false-positive warnings from ESP-IDF internals
  Suppress:
    - -Wno-unused-parameter

Index:
  Background: Build          # index in the background, don't block the agent
```

## 6. OpenCode LSP Configuration

### `opencode.json` — project-level override

```jsonc
{
  "$schema": "https://opencode.ai/config.json",

  "lsp": {
    // Override the built-in "clangd" entry with the ESP-IDF clangd
    "clangd": {
      "command": [
        // Replace with your actual esp-clang version path:
        "/Users/YOUR_USER/.espressif/tools/esp-clang/esp-18.1.2_20240912/esp-clang/bin/clangd",
        "--background-index",
        "--clang-tidy",
        "--completion-style=detailed",
        // Cross-compiler — pick the right one for your target:
        // ESP32 / S2 / S3 (Xtensa):
        "--query-driver=/Users/YOUR_USER/.espressif/tools/xtensa-esp-elf/esp-14.2.0_20241119/xtensa-esp-elf/bin/xtensa-esp*-elf-gcc",
        // ESP32-C3/C6/H2 (RISC-V) — use this instead or in addition:
        // "--query-driver=/Users/YOUR_USER/.espressif/tools/riscv32-esp-elf/.../riscv32-esp-elf-gcc",
        "--compile-commands-dir=build"
      ],
      "extensions": [".c", ".cpp", ".cc", ".h", ".hpp"]
    }
  }
}
```

### `opencode-shared` — add to the init script

Since you use git submodules / symlinks for shared agent config, add a helper
that resolves the actual `esp-clang` path at init time so the version string
doesn't go stale:

```bash
# In opencode-shared/init.sh (add after existing symlink setup)

if [ -f "$IDF_PATH/export.sh" ]; then
  echo "[init] Detected ESP-IDF project, generating ESP32 opencode.json..."

  # Resolve current esp-clangd binary
  ESP_CLANGD=$(ls ~/.espressif/tools/esp-clang/*/esp-clang/bin/clangd 2>/dev/null | tail -1)
  XTENSA_GCC=$(ls ~/.espressif/tools/xtensa-esp-elf/*/xtensa-esp-elf/bin/xtensa-esp32-elf-gcc 2>/dev/null | tail -1)

  if [ -n "$ESP_CLANGD" ] && [ -n "$XTENSA_GCC" ]; then
    XTENSA_DIR=$(dirname "$XTENSA_GCC")
    cat > opencode.json <<EOF
{
  "\$schema": "https://opencode.ai/config.json",
  "lsp": {
    "clangd": {
      "command": [
        "$ESP_CLANGD",
        "--background-index",
        "--clang-tidy",
        "--completion-style=detailed",
        "--query-driver=${XTENSA_DIR}/xtensa-esp*-elf-gcc",
        "--compile-commands-dir=build"
      ],
      "extensions": [".c", ".cpp", ".cc", ".h", ".hpp"]
    }
  }
}
EOF
    echo "[init] opencode.json written with esp-clangd at $ESP_CLANGD"
  else
    echo "[init] WARNING: esp-clangd or xtensa-gcc not found — LSP disabled"
  fi
fi
```

## 7. `AGENTS.md` — Wiring LSP into the Agent Loop

This is how you close the loop between LSP diagnostics and your existing agent modes.

```markdown
# ESP32 Project — Agent Instructions

## Environment
- IDF_PATH: ~/esp/esp-idf  (source export.sh before any build command)
- Target: esp32   (or esp32c3 for RISC-V)
- Build dir: ./build

## Toolchain commands (CLI truth > LSP for final verification)
- Reconfigure (refresh compile_commands.json): `idf.py reconfigure`
- Build:    `idf.py build 2>&1 | tee build.log`
- Flash:    `idf.py -p /dev/cu.usbserial-* flash`
- Monitor:  `idf.py monitor`
- Clean:    `idf.py fullclean`

## LSP usage rules
- clangd is active for .c/.cpp/.h files — use diagnostics as a FIRST PASS signal
- LSP errors about missing symbols in esp-idf/components/** are likely false positives
  if compile_commands.json is stale — run `idf.py reconfigure` to refresh it
- Always confirm errors with `idf.py build` before concluding something is broken

## Agent mode routing for ESP32 tasks
| Task                             | Agent  | Primary signal       |
|----------------------------------|--------|----------------------|
| Explore component API            | scout  | LSP hover + qmd      |
| Design a new driver / component  | plan   | qmd memory           |
| Implement / edit source files    | dev    | LSP diagnostics      |
| Verify correctness               | qa     | idf.py build output  |
| Save patterns to memory          | mem    | qmd store            |

## Debugging hints for `dev` agent
1. Check LSP diagnostics first (no build required)
2. If diagnostics are clean, run `idf.py build` and parse build.log
3. For linker errors: grep build.log for "undefined reference"
4. For flash/runtime: `idf.py monitor` output is ground truth
```

## 8. Agent Mode Adjustments

### `dev` agent — add LSP-aware behavior

In `.opencode/agent/dev.md` (or your shared copy), add to the system prompt:

```markdown
## LSP Feedback Loop
When editing C/C++ files in an ESP-IDF project:
1. After each file edit, wait for LSP diagnostics to update (they appear within ~2s)
2. Fix all ERROR-level diagnostics before moving to the next file
3. Treat WARNING-level diagnostics as informational unless they are in code you wrote
4. If an include cannot be resolved and compile_commands.json exists, the issue is
   likely a missing `#include` — do not delete the include, investigate the path
5. Run `idf.py build` as final confirmation before marking a task done
```

### `qa` agent — add build verification step

```markdown
## ESP32 Build Verification
Before approving any change:
1. Run `idf.py reconfigure` to ensure compile_commands.json is current
2. Run `idf.py build 2>&1 | tee build.log`
3. Check build.log for: ERROR, WARNING, undefined reference, multiple definition
4. Report: binary size, free heap (if visible in map file), any new warnings
```

### `mem` agent — ESP32 patterns worth memorizing

Prompt the `mem` agent to store:
- Successful FreeRTOS task patterns with correct stack sizes
- Working SPI/I2C component configurations
- Partition table layouts that worked
- Flash frequency + mode combinations for your specific ESP32 variant

## 9. `check.sh` Health Additions

```bash
# Add to opencode-shared/check.sh

echo "=== ESP32 / LSP Health ==="

# Check esp-clangd
ESP_CLANGD=$(ls ~/.espressif/tools/esp-clang/*/esp-clang/bin/clangd 2>/dev/null | tail -1)
if [ -n "$ESP_CLANGD" ]; then
  echo "✅ esp-clangd: $ESP_CLANGD"
else
  echo "❌ esp-clangd not found — run: idf_tools.py install all"
fi

# Check compile_commands.json
if [ -f "build/compile_commands.json" ]; then
  AGE=$(( ($(date +%s) - $(stat -f %m build/compile_commands.json)) / 60 ))
  echo "✅ compile_commands.json exists (${AGE}m old)"
  if [ "$AGE" -gt 60 ]; then
    echo "⚠️  compile_commands.json is >1h old — consider idf.py reconfigure"
  fi
else
  echo "❌ compile_commands.json missing — run: idf.py reconfigure"
fi

# Check .clangd config
if [ -f ".clangd" ]; then
  echo "✅ .clangd project config present"
else
  echo "⚠️  .clangd not found — clangd will emit GCC-flag warnings"
fi

# Check IDF_PATH
if [ -n "$IDF_PATH" ]; then
  echo "✅ IDF_PATH=$IDF_PATH"
else
  echo "❌ IDF_PATH not set — source \$IDF_PATH/export.sh"
fi
```

## 10. Implementation Sequence

```
Phase 1 — Foundation (30 min)
  ├─ Install esp-clang via idf_tools.py install all
  ├─ idf.py reconfigure  →  build/compile_commands.json
  └─ Create .clangd with flag removals

Phase 2 — OpenCode wiring (15 min)
  ├─ Write opencode.json with esp-clangd path
  ├─ Open a .c file in opencode → confirm clangd starts (check logs)
  └─ Verify diagnostics are clean on a known-good file

Phase 3 — Agent integration (20 min)
  ├─ Update AGENTS.md with CLI commands + LSP routing table
  ├─ Add LSP feedback loop instructions to dev.md
  └─ Add build verification block to qa.md

Phase 4 — Shared repo update (10 min)
  ├─ Add ESP-IDF detection + opencode.json generator to init.sh
  ├─ Add ESP32 health checks to check.sh
  └─ Commit to opencode-shared, pull into project via submodule

Phase 5 — Validation
  ├─ Introduce a deliberate type error → confirm agent sees LSP diagnostic
  ├─ Fix it via dev agent → confirm LSP clears, build passes
  └─ Run mem agent to store the pattern
```
